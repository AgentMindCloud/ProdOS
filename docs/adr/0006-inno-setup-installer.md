# ADR 0006: Inno Setup installer, per-user install, windowed build

## Status
Accepted

## Context
The initial Windows packaging work (see
[ADR 0005](0005-pyinstaller-onedir-frozen-path-resolution.md)) produced a
PyInstaller onedir bundle distributed as a zip: download, extract, find
`ProducerOS.exe` inside a folder, double-click it, and see a console
window while it runs. That's a fine developer/portable workflow, but it
fails the actual goal for a non-technical end user: no installer means no
Start Menu entry, no desktop icon, no listing in "Apps & features" to
uninstall from, and no clean upgrade story beyond "delete the old folder
and extract the new one over it." A console window flashing open also
reads as "technical/scary," not as a normal app launching.

Three real options for producing an actual installer: WiX Toolset (MSI,
the "enterprise" standard, steep authoring curve, XML-heavy), NSIS
(scriptable, widely used, somewhat dated syntax), and Inno Setup
(scriptable, free, Pascal-Script for advanced logic, `windows-latest`
GitHub Actions runners ship it preinstalled). A single admin-vs-per-user
install decision also had to be made: an admin install to
`C:\Program Files\` triggers a UAC "allow this app to make changes?"
prompt on every install and every update, and fails outright for a user
without admin rights on their own machine.

## Decision
**Inno Setup**, scripted at `packaging/inno/producer-os.iss`, built via
`scripts/build_installer.ps1` (wraps the existing PyInstaller build, then
compiles the installer) and in CI via `.github/workflows/windows-build.yml`
/ `release.yml`. Chosen over WiX/MSI for authoring simplicity relative to
what ProducerOS actually needs (no enterprise deployment features, no
Group Policy integration, no MSI transform requirements) and because
`windows-latest` already has it, so CI needs no extra toolchain setup
(a Chocolatey fallback install step exists in case a future runner image
drops it). Chosen over NSIS mainly for Inno's more approachable,
better-documented Pascal Script when custom logic (see the
post-uninstall data-preservation message in the `.iss` file) is needed.

**Per-user install** (`PrivilegesRequired=lowest`,
`DefaultDirName={localappdata}\Programs\ProducerOS`): no UAC prompt, no
admin rights required, matching how VS Code, Discord, and Slack install
themselves. This directly serves the "as easy as possible for a
non-technical person" goal -- there is no dialog to be confused by beyond
the installer wizard itself.

**A fixed `AppId` GUID**, embedded permanently in the `.iss` file. Windows
uses this, not the app name or version, to recognize "this is the same
application" across installer runs -- running a newer version's installer
with the same `AppId` upgrades the existing install in place (same
shortcuts, registered in "Apps & features" as one entry with an updated
version number) rather than creating a second, parallel install. This
GUID must never change across any future ProducerOS release.

**The install directory and the data directory are different paths on
purpose**: the app installs to
`%LOCALAPPDATA%\Programs\ProducerOS`, while ProducerOS's actual data
(database, logs, backups, secret key -- see `docs/ADMIN_GUIDE.md`) lives
at `%LOCALAPPDATA%\ProducerOS`. Because they don't overlap, upgrading
(the installer's `[InstallDelete]` step wipes and replaces the bundled
`_internal\` runtime on every install) or uninstalling the app never
touches user data -- no special-case data-preservation code was needed
beyond keeping these two paths distinct, which was already true before
this ADR.

**Windowed PyInstaller build** (`console=False` in `produceros.spec`,
switched from the earlier `console=True`): double-clicking the desktop
icon should feel like launching a normal app, not a terminal program.
Startup failures still surface via a Windows message box
(`packaging/pyinstaller/launcher.py`, already existed for this purpose),
and every log line that would have gone to a console still reaches the
on-disk log file (`logging_config.py` already writes both). This surfaced
one real bug while making the change: a windowed PyInstaller build gets
`sys.stdout`/`sys.stderr` of `None` from Windows (there is no console to
attach to), which crashes on the very first bare `print()` or the
logging module's default `StreamHandler`. `launcher.py`'s
`_fix_windowed_stdio()` redirects both to a null sink before anything
else runs, fixing this without needing to touch every call site.

**A real Windows `.ico`**, generated by extending the existing pure-Python
PNG generator (`scripts/generate_icons.py`) rather than adding an image
library dependency: modern `.ico` files can embed PNG-compressed frames
directly (supported since Windows Vista), so the same PNG encoder already
used for the PWA icons packs into a valid multi-resolution `.ico` with a
short, dependency-free function. This `.ico` is embedded into the `.exe`
itself (`EXE(icon=...)` in `produceros.spec`) so the taskbar, Start Menu,
and desktop shortcut all show the real ProducerOS mark instead of
PyInstaller's generic icon.

## Consequences
- A non-technical user's actual workflow is: download one `.exe` from
  GitHub Releases, double-click it, click through a wizard with no admin
  prompt, get a desktop icon. Updating is: download the new `.exe`, run
  it the same way. Nothing about this requires them to understand zip
  files, folder structures, or PATH.
- The portable zip distribution from ADR 0005 is kept as a secondary
  release asset for anyone who explicitly wants a no-installer option,
  but the installer is now the documented, recommended, headline
  download (`docs/INSTALL_WINDOWS.md`, the release notes template in
  `release.yml`).
- `windows-build.yml`'s smoke test now covers the full real-world path:
  silent-install the compiled installer, verify both shortcuts exist,
  launch the *installed* exe and confirm it serves real HTTP responses,
  then silently uninstall and confirm the data directory survives --
  not just "the raw PyInstaller bundle runs," which was all the previous
  smoke test checked.
- As with the PyInstaller `.exe` itself (ADR 0005), the `.iss` script has
  not been compiled on real Windows from this session -- there is no
  Windows machine or Inno Setup available in this build environment. It
  was written and manually reviewed against Inno Setup's documented
  syntax, but `windows-build.yml`'s CI run on a real `windows-latest`
  runner is what actually validates it compiles and behaves as designed.
  Treat it as syntax-reviewed, not build-verified, until that CI run is
  observed passing.
