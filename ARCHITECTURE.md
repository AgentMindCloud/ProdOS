# Architecture

How ProducerOS is put together and why. For the underlying product
constraints see `docs/PRODUCT_SPEC.md`; for individual decision records
see `docs/adr/`.

## Big picture

A single Python process serves everything: a FastAPI web application with
server-rendered Jinja2 templates (no SPA, no frontend build step, no CDN
assets), backed by a SQLite database in the user's data directory, plus an
optional MCP server thread. On Windows the whole thing freezes into one
PyInstaller onedir package with no separately-installed Python.

```
 Browser (desktop / paired Android phone)
        |
        v
 uvicorn -> FastAPI app (src/produceros/web/)
        |         routes/*  -- thin HTTP handlers: forms, CSRF, redirects
        v
 services/*  -- all business logic, one module per domain
        |
        v
 SQLAlchemy models (src/produceros/models/) -> SQLite (WAL) in the data dir
```

## Layering rules

- **`web/routes/*`** parse forms, verify CSRF, call a service, redirect.
  They contain no business logic beyond input shaping.
- **`services/*`** own all state changes. Every state-changing service
  call writes an `AuditEvent` (`services/audit.log_event`). Services take
  a `Session` as their first argument and never open their own except via
  the shared factories in `db/session.py`.
- **`models/*`** are declarative SQLAlchemy 2.x, dialect-agnostic (see
  ADR 0004), with all enums centralized in `models/enums.py`.
- **Domain packages** sit beside services for the heavier subsystems:
  `scanners/` (read-only filesystem scanning, filename parsing, hashing),
  `audio/` (Mutagen + stdlib `wave` fallback + optional FFmpeg),
  `marketing/` (deterministic local templates), `delivery/`
  (manifest -> approve -> execute packaging), `analytics/` (CSV import),
  `mcp_server/` (FastMCP tools), `demo/` (synthetic data with precise
  cleanup manifest).

## Key mechanisms

**Configuration** (`config.py`): zero-config defaults; overrides via
`PRODUCEROS_*` env vars or `config.toml` in the data directory (env wins).
The data directory (`%LOCALAPPDATA%\ProducerOS` on Windows) is the single
writable location -- database, secret key, logs, backups -- and is never
inside the install directory, which may be read-only when frozen.

**Auth** (`services/auth.py`, ADR 0003): Argon2 password hashing, signed
stateless session cookies, revocation via a per-user
`session_invalidated_before` timestamp instead of a session table.
LAN devices get their own signed cookie checked live against
`PairedDevice.status` on every request.

**CSRF** (`web/csrf.py` + middleware in `web/app.py`): double-submit
cookie. The token is issued by middleware onto `request.state.csrf_token`
*before* the route runs, because FastAPI discards `set_cookie()` calls
made on an injected `Response` parameter when the route returns its own
response object. New form routes must follow this pattern -- see
`docs/SECURITY_MODEL.md`.

**File safety** (`services/file_operations.py`): the scanner only
observes; any disk mutation is a proposed operation
(`PENDING_APPROVAL`, `dry_run=True`) that must be approved and is
re-validated against the allowed roots at execution time. `REPLACE` is
refused unconditionally.

**Migrations**: one Alembic revision; `migrations/env.py` resolves the DB
URL from settings at runtime (no hardcoded URL in `alembic.ini`), and the
CLI wraps Alembic's Python API so a frozen build migrates itself
(ADR 0005). CI runs `alembic check` to catch model/migration drift.

**Timestamps**: every datetime column uses the custom `UTCDateTime`
decorator in `db/base.py` -- SQLite returns naive datetimes for plain
`DateTime(timezone=True)` columns, which crashes tz-aware comparisons
(ADR 0004). Don't use raw `DateTime` in new models.

**Frontend**: one hand-written CSS file (`web/static/css/app.css`,
desktop sidebar / sub-768px bottom nav), one small progressive-enhancement
JS file, an SVG icon sprite, and PWA assets (manifest, service worker
that caches only the offline shell, icons generated by a pure-Python PNG
writer in `scripts/generate_icons.py`). No frameworks, no build step.

## Entry points

- `produceros run [--mode desktop|lan] [--port] [--no-browser]` -- runs
  migrations, then uvicorn; starts the MCP thread if `mcp_enabled`.
- `produceros db-upgrade | db-current | demo-load | demo-clean |
  backup-create | restore-dry-run | restore` -- see `cli.py`.
- Frozen Windows build: `packaging/pyinstaller/launcher.py` forwards argv
  into the same CLI (`packaging/README.md`).

## Testing strategy

- `tests/unit/` -- pure logic (parsers, checklist rules, security
  helpers, CLI wiring) against per-test isolated data dirs.
- `tests/integration/` -- real HTTP through `TestClient`: setup, login,
  workflows, backups, migrations.
- `tests/security/` -- CSRF, lockout, session invalidation, path
  traversal, shell-injection resistance, rate limiting, secret redaction.
- `tests/e2e/` -- a real uvicorn server driven by real Chromium
  (Playwright): the full producer workflow plus mobile-viewport layout
  assertions. This is the only layer that exercises the CSS/JS for real,
  and it has caught bugs the other layers structurally could not (e.g.
  the desktop list-visibility CSS bug).

Every test gets a fresh temp data directory via the fixtures in
`tests/conftest.py` -- tests never touch a developer's real data.
